# VCR0070: Success should not receive null

| Property    | Value                          |
|-------------|--------------------------------|
| **Rule ID** | VCR0070                        |
| **Category**| Usage                          |
| **Severity**| Warning                        |
| **Enabled** | Yes                            |

## Description

`Result<T>.Success(null)` creates a successful result without a value, which defeats the purpose of the Result pattern. A successful result should always carry a meaningful value. Passing `null` leads to `NullReferenceException` when the value is used in railway chains (`Map`, `Bind`, `Tap`, `Ensure`).

## Examples

### Violating code

```csharp
// VCR0070: Success with null is an anti-pattern
return Result<RequiredFee?>.Success(null);

// Also flagged with default literal
return Result<Order?>.Success(default);

// Also flagged with cast null
return Result<string?>.Success((string?)null);

// Also flagged with nullable value types
return Result<int?>.Success(null);
```

### Fixed code

```csharp
// Option 1: Use Failure for absent values
return Result<RequiredFee>.Failure(Error.NotFoundError("Required fee not found"));

// Option 2: Use non-generic Result for void success
return Result.Success();

// Option 3: Return the actual value when it exists
var fee = CalculateFee(order);
return fee != null
    ? Result<RequiredFee>.Success(fee)
    : Result<RequiredFee>.Failure(Error.NotFoundError("No fee applies"));
```

## Why this is dangerous

`Success(null)` creates a time bomb in railway chains. The code appears to work until someone adds an operator:

```csharp
Result<RequiredFee?>.Success(null)
    .Map(fee => fee.Amount)    // 💥 NullReferenceException — fee is null
    .Bind(amount => Validate(amount));
```

It also forces nullable propagation through all downstream code:

```csharp
var fee = result.Value;      // RequiredFee? — nullable
var amount = fee?.Amount;    // decimal? — nullable propagation
var text = amount?.ToString(); // string? — and so on...
```

## Code fix

The code fix replaces `Success(null)` with `Failure(Error.NotFoundError("TODO: provide meaningful error"))`:

```csharp
// Before
var result = Result<Order?>.Success(null);

// After code fix
var result = Result<Order?>.Failure(Error.NotFoundError("TODO: provide meaningful error"));
```

## When to suppress

This rule should rarely be suppressed. If you have a legitimate case where `null` is a valid success value, suppress with a comment explaining why:

```csharp
#pragma warning disable VCR0070 // Null is valid: field is optional in this context
return Result<string?>.Success(null);
#pragma warning restore VCR0070
```

Consider instead modeling the absence explicitly with a dedicated type or using `Result` (non-generic) for operations with no return value.

